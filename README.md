# Zava Contracts

Soroban smart contracts powering the Zava ZK savings and credit system on Stellar.

---

## Overview

Zava uses two Soroban contracts that work together:

```
User deposits savings
        ↓
SAVINGS CONTRACT
        ← stores ZK commitments, builds Merkle tree
        ↓
User generates ZK proof (off-chain, in browser via Noir)
        ↓
VERIFIER CONTRACT
        ← verifies proof on-chain, emits credit signal
        ↓
Lender protocol reads credit signal → disburses loan
```

Neither contract ever stores a raw amount, wallet address, or identity. All sensitive data stays private. The contracts only see hashes, commitments, and proofs.

## Contract 1: Savings Contract

**Location:** `contracts/savings/src/lib.rs`

### What It Does

Accepts weekly savings deposits as cryptographic commitments. Each commitment is a Pedersen hash of the user's private secret and deposit amount — computed off-chain before submission. The contract builds a Merkle tree of all commitments so the ZK circuit can later prove inclusion.

### Storage

```rust
// Per deposit entry
pub struct Commitment {
    hash: BytesN<32>,       // Pedersen hash of (secret, amount)
    nullifier: BytesN<32>,  // hash(secret) — prevents replay
    week_number: u32,       // which savings period this belongs to
    timestamp: u64,         // ledger timestamp of deposit
}

// Contract-level state
pub struct SavingsState {
    commitments: Vec<Commitment>,      // all deposits ever made
    merkle_root: BytesN<32>,           // current Merkle root
    spent_nullifiers: Set<BytesN<32>>, // nullifiers already used
}
```

### Functions

#### `deposit(commitment: BytesN<32>, nullifier: BytesN<32>, week_number: u32)`

Records a new savings deposit.

**Parameters:**

- `commitment` — Pedersen hash of `(secret, amount)`, computed by the user off-chain
- `nullifier` — Pedersen hash of `(secret)`, used for replay protection
- `week_number` — which savings week this deposit belongs to (0-indexed)

**Checks:**

- Nullifier has not been spent before
- Week number is valid (no future weeks, no gaps larger than 2 weeks)
- Commitment is a valid 32-byte hash

**Effects:**

- Appends commitment to the commitments list
- Marks nullifier as spent
- Recomputes Merkle root
- Emits `DepositRecorded` event

**Example:**

```bash
stellar contract invoke \
  --id $SAVINGS_CONTRACT_ID \
  --fn deposit \
  --arg '{"commitment": "0xabc...", "nullifier": "0xdef...", "week_number": 0}' \
  --network testnet
```

---

#### `get_merkle_root() → BytesN<32>`

Returns the current Merkle root of all recorded commitments. The ZK circuit uses this as a public input when generating a proof — it proves that the user's private deposits are included in the on-chain tree without revealing which ones.

**Returns:** 32-byte Merkle root

---

#### `get_commitment_count() → u32`

Returns the total number of commitments recorded. Useful for the frontend to display savings streak length.

---

#### `is_nullifier_spent(nullifier: BytesN<32>) → bool`

Checks whether a nullifier has already been used. Called by the verifier contract and frontend before proof submission to detect replay attempts.

---

#### `get_commitments_by_range(start: u32, end: u32) → Vec<Commitment>`

Returns commitments for a range of week numbers. Used by the frontend to assemble the private inputs for proof generation.

---

### Events Emitted

| Event               | When                     | Data                       |
| ------------------- | ------------------------ | -------------------------- |
| `DepositRecorded`   | Every successful deposit | `week_number`, `timestamp` |
| `MerkleRootUpdated` | Every deposit            | `new merkle_root`          |

## Contract 2: Verifier Contract

**Location:** `contracts/verifier/src/lib.rs`

### What It Does

Accepts a Noir ZK proof and its public inputs, verifies the proof on-chain using the Groth16 verifier, and if valid, records a credit tier for the submitting wallet and emits a credit signal that lending protocols can consume.

### Storage

```rust
pub struct CreditRecord {
    wallet: Address,
    tier: CreditTier,         // Low / Medium / VeryLow
    verified_at: u64,         // ledger timestamp
    consistency_weeks: u32,   // what the proof claimed
    expires_at: u64,          // credit signals expire after 90 days
}

pub enum CreditTier {
    Medium,   // 8 consecutive weeks proven
    Low,      // 12 consecutive weeks proven
    VeryLow,  // 24 consecutive weeks proven
}
```

### Functions

#### `verify_proof(proof: Bytes, public_inputs: PublicInputs) → bool`

The core function. Verifies a Noir-generated ZK proof on-chain.

**Parameters:**

- `proof` — raw Groth16 proof bytes generated by the Noir circuit
- `public_inputs` — a struct containing:

```rust
pub struct PublicInputs {
    commitment_root: BytesN<32>,  // must match savings contract Merkle root
    min_weekly_amount: u64,       // minimum threshold claimed (in stroops)
    consistency_weeks: u32,       // number of consecutive weeks claimed
    nullifiers: Vec<BytesN<32>>,  // one per deposit period claimed
}
```

**Checks:**

- `commitment_root` matches the current Merkle root from the savings contract
- All nullifiers are unspent (cross-checks savings contract)
- Groth16 proof verifies against the Zava verification key
- `consistency_weeks` maps to a valid credit tier (8, 12, or 24)

**Effects (on success):**

- Records credit tier for the calling wallet
- Marks all submitted nullifiers as spent in the savings contract
- Sets `expires_at = current ledger time + 90 days`
- Emits `CreditVerified` event

**Reverts (on failure):**

- Invalid proof → `ProofInvalid`
- Root mismatch → `CommitmentRootMismatch`
- Spent nullifier → `NullifierAlreadySpent`
- Unknown tier → `InvalidConsistencyWeeks`

**Example:**

```bash
stellar contract invoke \
  --id $VERIFIER_CONTRACT_ID \
  --fn verify_proof \
  --arg '{"proof": "0x...", "public_inputs": {...}}' \
  --network testnet
```

---

#### `get_credit_tier(wallet: Address) → Option<CreditRecord>`

Returns the most recent valid credit record for a wallet, or `None` if no valid proof exists or the credit signal has expired.

Lending protocols call this to gate access or determine loan size.

**Returns:**

```rust
Some(CreditRecord {
    tier: CreditTier::Low,
    verified_at: 1234567890,
    consistency_weeks: 12,
    expires_at: 1242567890,
})
// or None
```

---

#### `is_credit_valid(wallet: Address) → bool`

Convenience function. Returns `true` if the wallet has a non-expired credit record. Lending protocols can use this as a simple gate check before `get_credit_tier`.

---

#### `get_verification_key() → Bytes`

Returns the Groth16 verification key embedded in the contract. Used by tooling and auditors to confirm the correct circuit is being verified.

---

### Events Emitted

| Event            | When                                        | Data                                                |
| ---------------- | ------------------------------------------- | --------------------------------------------------- |
| `CreditVerified` | Successful proof verification               | `wallet`, `tier`, `consistency_weeks`, `expires_at` |
| `CreditExpired`  | When a credit record is queried past expiry | `wallet`, `expired_at`                              |
| `ProofRejected`  | Failed verification attempt                 | `reason`                                            |

## Contract Interaction Flow

Complete flow from a developer perspective:

```rust
// 1. User computes commitment off-chain
commitment = pedersen_hash(secret, amount)
nullifier  = pedersen_hash(secret)

// 2. User calls savings_contract.deposit(commitment, nullifier, week_number)
//    → Contract stores commitment, updates Merkle root

// 3. After N weeks, user fetches:
merkle_root = savings_contract.get_merkle_root()
commitments = savings_contract.get_commitments_by_range(0, N)

// 4. User generates ZK proof off-chain (Noir WASM in browser):
proof = noir_circuit.prove({
  secret,
  weekly_amounts,
  deposit_timestamps,
  merkle_root,
  min_weekly_amount,
  consistency_weeks
})

// 5. User calls verifier_contract.verify_proof(proof, public_inputs)
//    → Contract verifies proof
//    → Emits CreditVerified event
//    → Records credit tier

// 6. Lending protocol calls:
verifier_contract.get_credit_tier(user_wallet)
//    → Reads credit tier
//    → Disburses loan based on tier
```

## Deployment

### Prerequisites

```
rust >= 1.70.0
stellar-cli >= 20.0.0
```

### Build

```bash
# Build both contracts
cd contracts
cargo build --target wasm32-unknown-unknown --release

# Output files:
# target/wasm32-unknown-unknown/release/savings_contract.wasm
# target/wasm32-unknown-unknown/release/verifier_contract.wasm
```

### Deploy to Testnet

```bash
# Deploy savings contract
stellar contract deploy \
  --wasm target/wasm32-unknown-unknown/release/savings_contract.wasm \
  --source $YOUR_SECRET_KEY \
  --network testnet

# → Returns SAVINGS_CONTRACT_ID, save this

# Deploy verifier contract (needs savings contract ID)
stellar contract deploy \
  --wasm target/wasm32-unknown-unknown/release/verifier_contract.wasm \
  --source $YOUR_SECRET_KEY \
  --network testnet

# Initialize verifier with savings contract address
stellar contract invoke \
  --id $VERIFIER_CONTRACT_ID \
  --fn initialize \
  --arg $SAVINGS_CONTRACT_ID \
  --network testnet
```

### Deploy to Mainnet

```bash
# Same as testnet — change --network flag
stellar contract deploy \
  --wasm target/wasm32-unknown-unknown/release/savings_contract.wasm \
  --source $YOUR_SECRET_KEY \
  --network mainnet
```

## Testing

```bash
cd contracts

# Run all tests
cargo test

# Run savings contract tests only
cargo test --package savings

# Run verifier contract tests only
cargo test --package verifier

# Run with output (useful for debugging)
cargo test -- --nocapture
```

### What the tests cover

**Savings contract:**

- Valid deposit accepted and commitment stored
- Duplicate nullifier rejected
- Merkle root updates correctly after each deposit
- Future week numbers rejected
- Gap detection (skipped weeks flagged)

**Verifier contract:**

- Valid proof accepted and credit tier recorded
- Invalid proof rejected with correct error
- Commitment root mismatch rejected
- Spent nullifier rejected
- Credit tier expiry works correctly
- Correct tier assigned for 8 / 12 / 24 week proofs

## Security Considerations

**Nullifier replay protection**
Every deposit generates a unique nullifier. Once spent in a proof, that nullifier can never be used again. This prevents a user from generating multiple credit proofs from the same savings history.

**Commitment root binding**
The ZK proof is cryptographically bound to the Merkle root at proof generation time. If the root changes between proof generation and submission (new deposit added), the proof is invalid. Users must generate proofs against a stable root.

**Credit expiry**
Credit signals expire after 90 days. Users must re-prove their savings behavior to maintain credit access. This ensures credit scores reflect current behavior, not a one-time historical snapshot.

**No admin keys**
Neither contract has an owner, admin, or upgrade key. Once deployed, the logic is immutable. Lending protocols can trust the credit signal without worrying about contract manipulation.

**Verification key immutability**
The Groth16 verification key is embedded at deployment time and cannot be changed. Any update to the ZK circuit requires deploying a new verifier contract, making circuit changes auditable and transparent.
